页

内核把物理页作为内存管理的基本单位，因为内存管理单元 MMU (Memory Management Unit) 这个硬件通常是以页为单位进行处理的。32 位的体系结构通常支持的是 4KB 的页大小，64 位体系结构通常支持的是 8KB 页大小。内核为每一个物理页都创建了一个这样的结构。

内核使用结构 page 来描述物理页（下面只选取了一些域，并且现在的 page 结构已经发生了变化，这里只为了能够说明，所以选择了 2.6 内核版本的页结构）：

  • flags ：该域用来存放页的状态，在文件 include/linux/page-flags.h 中详细的可表述的状态。flags 的每一个位都是一个状态，所以最多一次可以同时表示一个页的 32 种状态。
  • _count ：用来存放这个物理页的引用计数。当引用计数值减少为 -1 就表示当前内核没有引用该页，所以该页就可以用于新的分配（映射）。内核代码不要直接检查这个字段，而应该使用 page_count(struct page *) 接口来检查。 不过，该函数返回 0 表示当前页空闲；返回整数表示当前页正在被使用。
    一个页可以用于：页缓存、私有内存映射、进程页表中的映射等。
  • virtual ：表示的是页的虚拟地址。通常情况下这个域的值就是页在内核虚拟地址空间里面的虚拟地址。对于高端内存区域的页来说，virtual 域的值为 NULL，当内核需要操作高端内存区的页的时候，把它动态映射到内核虚拟地址空间的高端内存区里面，即 32 位 4G 内存机器下，虚拟地址范围为 3G+896M - 4G 的地址空间。

内核使用上述的 page 结构来管理内存中的所有页！

区

终于说到了区，有关于高端内存的解释就在后面。

首先说明以下几个事实：

  1. X86体系结构的 32 位机器上面，ISA 设备要执行 DMA(Directly Memory Access)，直接内存访问，只能寻址物理内存的前 16M。换句话说，只有物理内存的前 16M 可以用来执行 DMA 操作，其他的内存区域都不能。
  2. 不论是内核使用的页还是用户地址空间使用的页，内核都要负责管理，所以内核要能够感知到整个内存的页。
  3. 由于开启了分页机制，内核想要访问物理地址空间的时候，要先建立映射关系，通过虚拟地址来访问。
  4. 3G - 4G 的逻辑地址空间，如果都是有一个一一映射的关系，那么是不能映射 0 - 3G 的地址空间的，也就没有办法管理所有的页。

综合上面的四点，所以就有了内存分区得出现：

分区 描述 物理内存范围 虚拟内存范围
ZONE_DMA DMA 使用的页从这里分配 0-16M 3G-(3G+16M)
ZONE_DMA32 DMA 使用的页，但是只能被32位设备访问 0-16M 3G-(3G+16M)
ZONE_NORMAL 一般的内存需求可以从这里获取页 16M-896M (3G+16M)-(3G+896M)
ZONE_HIGHMEM 动态被映射的页 896M-1G (3G+896M)-4G


其中 ZONE_HIGHMEM 表示的就是那些不能被直接映射的地址空间，在 32 位机器上，代表的就是物理内存地址大于 896M 的地址，这些地址都应该动态的被映射到 (3G+896M)-4G 的地址空间上，所谓动态映射，举个例子，物理地址起始地址为 2G 的一个页被映射到了物理地址 896M-1G 上，此时的逻辑地址可能是 900M+PAGE_OFFSET，当该页被用完之后，解除了这种映射关系；另一个程序现在被加载到物理内存的 3G 地方，此时 3G 物理地址为起始地址的页就可以同样被映射到 900M+PAGE_OFFSET 的逻辑地址上，这就是动态映射，因为前面已经解除了 2G->900M+PAGE_OFFSET 的映射关系，所以这里 3G->900M+PAGE_OFFSET 的映射关系是可以被正确处理的。不过对于 64 位的机器来说，就没有 ZONE_HIGHMEM 区域，该区域在内核里面是空，因为 64 位的机器可以将至少 512G 的内存直接映射到 ZONE_NORMAL 区上面。

ZONE_DMA 的范围是 0-16M，该区域的物理页面专门供 I/O 设备的 DMA 使用。之所以要单独管理 DMA 页面，是因为 DMA使用物理地址直接访问内存，不经过 MMU，且需要连续的缓冲区。 所以为了提供物理上连续的缓冲区，必须从物理地址空间专门划分一段区域用于 DMA。
ZONE_NORMAL 的范围是 16M-896M，该区域的物理页面内核是可以直接使用的(直接将物理地址加上 0xC0000000 就得到了逻辑地址)。
ZONE_HIGHMEM 的范围是 896M-结束，该区域就是高端内存，内核不能直接使用，需要经过映射，转换为逻辑地址。

如前面的第三张图所示的那样，内核空间高端的 128M 又主要分为三部分：vmalloc area、持久内核映射区、临时内核映射区。

由于 ZONE_NORMAL 和内核线性空间存在直接映射关系，所以内核会将频繁使用的数据，如 kernel 代码、GDT、IDT、PGD、mem_map 数组，这些放在 ZONE_NORMAL 区里面，而将 用户数据、页表 等不常用数据放在 ZONE_HIGHMEM 里面，只在要访问这些数据的时候才建立映射关系（通过 kmap() 函数），使用完之后就断开映射关系。

由于开启了分页机制，内核想要访问物理地址空间的话，必须先建立映射关系，然后通过虚拟地址来访问。为了能够访问所有的物理地址空间，就要将全部物理地址空间映射到1G的内核线性空间中，这显然不可能。于是，内核将0~896M的物理地址空间一对一映射到自己的线性地址空间中，这样它便可以随时访问ZONE_DMA和ZONE_NORMAL里的物理页面；此时内核剩下的128M线性地址空间不足以完全映射所有的ZONE_HIGHMEM，Linux采取了动态映射的方法，即按需的将ZONE_HIGHMEM里的物理页面映射到kernel space的最后128M线性地址空间里，使用完之后释放映射关系，以供其它物理页面映射。虽然这样存在效率的问题，但是内核毕竟可以正常的访问所有的物理地址空间了。

上述四段话摘录自博客《linux内核内存管理(zone_dma zone_normal zone_highmem)》。

内核将高端内存划分为3部分：VMALLOC_STARTVMALLOC_END、KMAP_BASEFIXADDR_START和FIXADDR_START~4G。

对于高端内存，可以通过 alloc_page() 或者其它函数获得对应的 page，但是要想访问实际物理内存，还得把 page 转为线性地址才行（为什么？想想 MMU 是如何访问物理内存的），也就是说，我们需要为高端内存对应的 page 找一个线性空间，这个过程称为高端内存映射。

对应高端内存的3部分，高端内存映射有三种方式：

  1. 映射到“内核动态映射空间”（noncontiguous memory allocation）
    这种方式很简单，因为通过 vmalloc() ，在”内核动态映射空间”申请内存的时候，就可能从高端内存获得页面（参看 vmalloc 的实现），因此说高端内存有可能映射到”内核动态映射空间”中。

  2. 持久内核映射（permanent kernel mapping）
    如果是通过 alloc_page() 获得了高端内存对应的 page，如何给它找个线性空间？
    内核专门为此留出一块线性空间，从 PKMAP_BASE 到 FIXADDR_START ，用于映射高端内存。在 2.6内核上，这个地址范围是 4G-8M 到 4G-4M 之间。这个空间起叫”内核永久映射空间”或者”永久内核映射空间”。这个空间和其它空间使用同样的页目录表，对于内核来说，就是 swapper_pg_dir，对普通进程来说，通过 CR3 寄存器指向。通常情况下，这个空间是 4M 大小，因此仅仅需要一个页表即可，内核通过来 pkmap_page_table 寻找这个页表。通过 kmap()，可以把一个 page 映射到这个空间来。由于这个空间是 4M 大小，最多能同时映射 1024 个 page。因此，对于不使用的的 page，及应该时从这个空间释放掉（也就是解除映射关系），通过 kunmap() ，可以把一个 page 对应的线性地址从这个空间释放出来。

  3. 临时映射（temporary kernel mapping）
    内核在 FIXADDR_START 到 FIXADDR_TOP 之间保留了一些线性空间用于特殊需求。这个空间称为”固定映射空间”在这个空间中，有一部分用于高端内存的临时映射。
    这块空间具有如下特点：

    1. 每个 CPU 占用一块空间
    2. 在每个 CPU 占用的那块空间中，又分为多个小空间，每个小空间大小是 1 个 page，每个小空间用于一个目的，这些目的定义在 kmap_types.h 中的 km_type 中。

    当要进行一次临时映射的时候，需要指定映射的目的，根据映射目的，可以找到对应的小空间，然后把这个空间的地址作为映射地址。这意味着一次临时映射会导致以前的映射被覆盖。通过 kmap_atomic() 可实现临时映射。

上述内容摘录自博客《Linux内核高端内存》。

注意，区的划分没有任何物理意义，这只不过是内核用来管理页而采取的一种逻辑分组。 某些分配可能需要从特定的区中获取页，不过有的页可以从多个区中获得。例如用于 DMA 操作的页只能从 DMA 区获取，而用于一般操作的页，既可以从 ZONE_DMA 中获得，也可以从 ZONE_NORMAL 区中获得。如果一个区的可用空间不够了，那么内核会去占用别的可用区的内存。

获得页与释放页（以页为单位来分配及释放）

获得页

页分配函数 描述
struct page* alloc_pages(gfp_mask,order) 分配 2order个连续物理页，返回第一个页的 page 结构指针
struct page* alloc_page(gfp_mask) 只分配一页，返回指向描述物理内存页的页结构指针
struct page*page_address(&a_page) 返回 a_page 所指物理页的逻辑地址
unsigned long __get_free_pages(gfp_mask,order) 分配 2order个连续物理页，返回第一个页的逻辑地址
unsigned long __get_free_page(gfp_mask) 只分配一页，返回该页所在的逻辑地址
unsigned long get_zeroed_page(gfp_mask) 只分配一页，返回该页所在的逻辑地址，并将页内数据都填充为 0

  • struct page* alloc_pages(gfp_t gfp_mask,unsigned int order)
    该函数分配 2order 个连续物理页，并返回一个指针，该指针指向第一个页的 page 结构；如果出错，就返回 NULL。
  • struct page*page_address(struct page* a_page)
    该函数返回一个指针，指向给定的物理页当前所在的逻辑地址。
  • unsigned long __get_free_pages(gfp_t gfp_mask,unsigned int order)
    这个函数与 alloc_pages() 作用相同，不过它直接返回所请求的第一个页的逻辑地址。因为页是连续的，所以其他页也会紧随其后。
  • struct page* alloc_page(gfp_t gfp_mask)
  • unsigned long __get_free_page(gfp_t gfp_mask)
    上面两个函数都是只请求分配一个页，所以相较于带 ‘s’ 的版本，就没有 order 参数。
  • unsigned long get_zeroed_page(gfp_t gfp_mask)
    这个函数与 __get_free_page() 工作方式相同，不过会把分配好的页内的数据都填充为 0。

释放页

页释放函数 描述
void __free_pages(&a_page,order) 释放 a_page 所指向的页结构体，order 与页分配函数中的意义相同
void free_pages(addr,order) 释放逻辑地址为 addr 的页，order 与页分配函数中的意义相同
void free_page(addr) 释放逻辑地址为 addr 的页

  • void __free_pages(struct page* a_page,unsigned int order)
    只能释放属于你自己的页！如果传递了错误的 page 结构或者地址，或者用了错误的 order 数值，都可能导致系统崩溃！ 这个函数释放 2order 个连续的物理页，以 a_page 为计算页数的起始页。
  • void free_pages(unsigned long addr,unsigned int order)
    只能释放属于你自己的页！如果传递了错误的 page 结构或者地址，或者用了错误的 order 数值，都可能导致系统崩溃！ 这个函数释放 2order 个连续的物理页，以 addr 逻辑地址所在物理页为计算页数的起始页。
  • void free_page(unsigned long addr)
    只能释放属于你自己的页！如果传递了错误的 page 结构或者地址，或者用了错误的 order 数值，都可能导致系统崩溃！ 这个函数释放 addr 逻辑地址所在物理页。

以字节位单位来分配及释放

kmalloc()

kmalloc() 函数与用户空间的 malloc() 函数非常相似，但是 kmalloc() 多了一个参数 flags。

该函数用于以字节为单位申请内核内存。如果要申请的内存是一个页，那么还是使用上一节所提到的页分配接口。该函数返回的是一个指向大小至少为 size 的内存块的指针，这个内存块在物理上是连续的，指针值是逻辑地址。如果分配出错，返回的是 NULL。除非是没有足够的内存可用，否则总是会分配成功的。因为可能会分配出错，因此对 kmalloc 的返回值进行非空判断也是必须的。

gfp_t 标志

不管是在低级的页分配函数中，还是在kmalloc()这个按字节分配的函数中，都要求提供分配器标志。

分配器标志分为三类：行为修饰符 、区修饰符 、类型 。一般来说，类型 类别是对行为修饰符和区修饰符的组合，所以一般来说，我们直接使用类型类的分配器标志就可以了。不过下面还是简单介绍一下前两种。

  • 行为修饰符
    表示内核应当如何分配所需的内存。是否可睡眠？是否可以启动磁盘？是否可以启动文件系统等。
    标志 描述
    __GFP_WAIT 分配器可以睡眠
    __GFP_HIGH 分配器可以访问紧急事件缓冲池
    __GFP_IO 分配器可以启动磁盘 I/O
    __GFP_FS 分配器可以启动文件系统 I/O
    __GFP_COLD 分配器应该使用高速缓冲中快要淘汰出去的页
    __GFP_NOWARN 分配器将不打印失败警告
    __GFP_REPEAT 分配器在分配失败时重新进行分配，但是重新进行的分配还是可能失败
    __GFP_NOFALL 如果分配失败，分配器将无限的重复进行分配
    __GFP_NORETRY 分配器在分配失败的时候绝不重新分配
    __GFP_NO_GROW 由 slab 内部使用
    __GFP_COMP 添加混合页元数据，在 hugeltb 的代码内部使用
  • 区修饰符
    区修饰符表示分配器该从哪里分配内存。通常，分配器可以从任何区中分配内存，不过内核优先从 ZONE_NORMAL 区中分配，以保证别的区在需要内存的时候有足够的空闲页可用。
    标志 描述
    __GFP_DMA 从 ZONE_DMA 区分配
    __GFP_DMA32 只在 ZONE_DMA32 区分配
    __GFP_HIGHMEM 从 ZONE_HIGHMEM 或者 ZONE_NORMAL 区分配
    - __GFP_DMA 标志强制内核从 ZONE_DMA 区去分配，所以得到的一定是可以用于 DMA 的内存。
    - 指定了 __GFP_HIGHMEM 会优先从 ZONE_HIGHMEM 区得到分配，如果不能获得有效的期望分配，那么会从 ZONE_NORMAL 区得到分配。
    - 如果没有指定任何区修饰符，那么会优先从 ZONE_NORMAL 区得到分配，如果不能获得有效的期望分配，那么会从 ZONE_DMA 区得到分配。
    - 此外还有一点需要注意，不能给 __get_free_pages() kmalloc() 指定 ZONE_HIGHMEM，因为这两个函数返回的都是内核空间逻辑地址，而不是 page 结构。有可能这两个函数所分配的内存(地址大于 896M)，还没有映射到内核的虚拟地址空间，所以在内核空间也就没有与之对应的逻辑地址。

为了避免自己组装上面的修饰符的时候出错，所以内核提供了 类型，即，对修饰符组装之后的标志，来帮助分配器分配合适的内存。

类型 描述 修饰符
GFP_ATOMIC 这个标志用在中断处理程序、下半部、持有自旋锁以及其他不能睡眠的地方 __GFP_HIGH
GFP_NOWAIT 与 GFP_ATOMIC 类似，不过他不会将分配推给紧急内存池，增加了内存分配失败的可能性 0
GFP_NOIO 这种分配可以阻塞，但是不会启动磁盘 I/O。这个标志在不能引发更多磁盘 I/O 时阻塞 I/O 代码，这可能导致不好的递归。 __GFP_WAIT
GFP_NOFS 这种分配在必要时可能阻塞，也可能启动磁盘 I/O，但是不会启动文件系统操作。这个标志在不能再启动另一个文件系统的操作时，用在文件系统部分的代码中 (GFP_WAIT|GFP_IO)
GFP_KERNEL 常规的分配方式，可能会阻塞。这个标志在睡眠安全时用在进程上下文代码中，为了获得调用者所需的内存，内核会尽力而为。这个标识应当是首选项。 (GFP_WAIT|GFP_IO|__GFP_FS)
GFP_USER 这也是一种常规分配方式，可能会阻塞。这个标志用于为用户空间的进程分配内存。 (GFP_WAIT|GFP_IO|__GFP_FS)
GFP_HIGHUSER 这是从 ZONE_HIGHMEM 进行分配，可能会阻塞。这个标志用于为用户空间进程分配内存 (GFP_WAIT|GFP_IO|GFP_FS|GFP_HIGHMEM)
GFP_DMA 从 ZONE_DMA 进行分配。这个标志通常与上面的某个标志组合使用 (__GFP_DMA)

上述的 GFP_ATOMIC 与 GFP_KERNEL 这两个常用的类型，应当重点被区分开。最主要的就是 GFP_ATOMIC 用在不能睡眠的场景，而 GFP_KERNEL 用在可睡眠场景，并且内核尽最大努力分配。 所以综合来看，因为使用 GFP_ATOMIC 使用的场景不能睡眠，所以当当前系统内存不够的时候，也没有办法让当前的进程先睡眠，然后调用别的进程来释放掉一些内存，就导致了使用 GFP_ATOMIC 在内存紧张的时候是很容易分配失败的。而 GFP_KERNEL 在内存紧张的时候，可以先睡眠，然后调用交换刷新一些内存到硬盘，这样就就有空闲内存来提供给此次请求，所以 GFP_KERNEL 分配的成功几率会比 GFP_ATOMIC 高。

至于会出现 GFP_NOFS 和 GFP_NOIO，这是因为在一些低级的内存分配场景中，如 IO 设备和文件系统里面，如果在他们的代码里面还会去请求 I/O 操作或者是文件系统的操作，那么这些操作又会带来新的内存请求，这样递归下去，就会导致系统出错。所以这些代码在被调用的时候，必须保证分配器不会再执行到其本身，否则分配可能会产生死锁。

不过在内核代码的编写中，基本上用到的也就是 GFP_ATOMIC 和 GFP_KERNEL。

vmalloc()

该函数返回一个指针，指向的是逻辑上连续的一块内存，大小至少为 size。发生错误的时候，函数返回 NULL。函数可以睡眠，不能在中断上下文中调用，也不允许在其他不允许阻塞的情况下调用。

使用 vfree() 来释放由 vmalloc() 申请的内存，该函数可以睡眠，没有返回值。

vmalloc() 的工作方式和 kmalloc() 是类似的，不过 kmalloc() 分配的内存不仅在逻辑上是连续的（虚拟地址连续），在物理上也是连续的；但是 vmalloc() 所分配的内存，其虚拟地址是连续的，但是物理上无须连续。vmalloc() 通过分配非连续的物理内存块，然后“修正”页表，把内存映射到逻辑地址空间的连续区域中。

大多数情况下，只有硬件设备需要用到连续得物理内存。在很多体系结构上，因为硬件设备是不在内存管理单元的管理范围之内，所以这些硬件根本就不知道虚拟地址的存在，也不会去使用虚拟地址（因为没有 MMU 来执行从虚拟地址到物理地址的翻译）。而软件使用的就是逻辑地址，因为有 MMU 来完成地址翻译，所以使用逻辑地址和虚拟地址对软件来说没有区别，再加上之前对 ZONE_HIGHMEM 的讨论，内核空间只有 1G，要管理整个内存，也需要使用虚拟地址来做动态映射。

不过出于对性能的考虑，内核代码里面常用的还是 kmalloc()。因为使用 vmalloc() 把不连续的物理地址映射到连续的逻辑地址空间，需要专门建立页表，而通过 vmalloc() 所申请的多个不连续物理页，需要一页一页的进行逻辑地址的映射，这就比申请的连续物理页进行直接内存映射（多个页所加的偏移值相同，甚至可以直接使用位运算）所造成的 TLB 抖动大得多。所以在内核代码里面较少会使用 vmalloc()，除非是需要申请大块内存，例如把模块动态插入到内核里面，这时候就把模块加载到由 vmalloc() 申请的内存上。

伙伴系统(buddy system)

linux 内核采用了一种同时适用于 32 位和 64 位的内存分页模型。对于 32 位的系统来说，两级页表就足够寻址 32 位地址；而对于 64 位系统来说，需要使用四级页表：

  • PGD 页全局目录 (Page Global Directory)
  • PUD 页上级目录 (Page Upper Directory)
  • PMD 页中间目录 (Page Middle Directory)
  • PT     页表 (Page Table)

其中，PGD 的一个目录项就是一个 PUD 的地址，一个 PUD 目录项就是一个 PMD 地址，当然最后一级就是页表项，每个页表项指向一个 4KB 大小的页框，4KB 大小的页框是标准的内存分配单元。

在实际应用中，经常需要分配一组连续的页框，而频繁地申请和释放不同大小的连续页框，必然导致在已分配页框的内存块之间分散了许多小块的空闲页框。这样，即使这些页框是空闲的，其总大小是足够一次应用请求的，但是由于不连续，所以可能对要求申请连续页框的应用难以得到满足。

为了尽量避免上面的情况出现，linux 内核引入了伙伴系统(Buddy System)，其原理类似于 STL 里面的二级分配器，使用了内存池的方式：

内核把所有的空闲页框分组成 11 个链表，每个链表上的元素对应的不同页框大小的连续内存块，20~10 个页框大小，即，第一个链表上的元素都是 1（20）个页框大小的空闲内存块，第 7 条链表上的元素都是 64（26）个页框大小的连续空闲内存块，以此类推。最大可以申请 1024 个连续页框，1024 个连续页框对应的是 4MB 大小的连续内存。

假如要申请一个 256 连续页框大小的内存块，首先会从第 9（28） 条链表上去找空闲块，如果有则将其从空闲块上摘下，返回个请求者；如果没有，就去其后面的链表上去找，如果找到了，就将找到的空闲块，每次取一半，接到对应的链表上，然后返回满足要求的最小块给申请者。要是 11 条空闲链表都没有空闲块，那么就返回错误。

页框块在释放的时候，会主动将两个连续的页框块合并为一个较大的页框块。

slab 层

在没有 slab 的情况下，那些内核需要频繁申请和释放的对象会对内存造成负担，一方面是这些对象是内核数据结构，比较小，频繁的分配和释放会产生碎片；另一方面，在使用的时候再从内存去分配，用完之后又归还给内存，在频繁使用的情境下，这样是比较耗时的。所以就出现了 slab，他是在低级页分配的基础上再抽象出来的一层，管理的是高速缓存。

在 slab 中，有三层管理结构：

  1. slab 分配器管理的是高速缓存组；
  2. 高速缓存组管理的 slab 组；
  3. slab 对象管理的是内核数据对象组。

注意，slab 分配器是最高层的管理策略，而下面的 slab 组是 slab 对象数组，slab 对象是一个（或者多个）页框，一个页框里面有多个内核数据对象。一个 slab 对象有三种状态：满的，空的，部分满。

在栈上的静态分配

这里的栈指的是内核栈，所谓在内核栈上静态分配指的是在一个内核函数里面创建自动变量。再讨论在栈上创建自动变量之前，先来说一下什么是内核栈，什么是用户栈，以及他们的主要区别。

下面的内容大多摘录自博客《进程的用户栈和内核栈》，在其中我根据我上面讲述的内容做了一些额外补充说明。

一个进程有两个堆栈，用户堆栈和内核堆栈。内核在创建一个新的进程的时候，创建进程描述符 task_struct 对象的同时，也会为其分配好堆栈（我的个人注解– 所分配好的应该是内核堆栈，用户堆栈是可以动态增长的，所以一开始应该是一个基址。 –我的个人注解）。用户堆栈的空间指向的是用户地址空间，内核堆栈的空间指向的是内核地址空间。当进程运行于用户态的时候，CPU 堆栈指针寄存器存放的是用户堆栈栈顶地址，使用用户栈；当进程运行于内核态的时候（或者叫当前内核代码运行于该进程上下文的时候），CPU 堆栈指针寄存器存放的是内核堆栈栈基址，使用内核栈。

当进程由于中断或者系统调用进入到内核态的时候，首先会将当前的用户堆栈栈顶地址保存到内核堆栈里面，然后设置 CPU 堆栈指针寄存器的值为：内核堆栈基址+一个地址，这样就完成了用户栈向内核栈的切换。（这里涉及到一个内核堆栈基址的问题，后面讲。）

当进程的内核代码执行完成之后，恢复保存在内核栈里面的用户栈栈顶地址到 CPU 栈指针寄存器，完成内核栈向用户栈的切换。

现在来说，为什么切换到内核栈的时候，是切换到：内核堆栈基址+一个地址。首先，当进程从用户态进入到内核态的时候，内核堆栈是空的。因为内核代码都是提供服务，切换到内核态是使用内核的服务，要是没有使用完服务，是不会退出回到用户态，所以每次进来出去，栈帧都会平衡到初始状态，所以下一次从用户态进入内核态，内核栈还是空的，所以内核栈的切换只要换到内核栈的起始位置就好了，不过压了一个用户栈地址到内核栈，所以就变成了上面使用的地址。

说到了这里，结合之前说的内核页分配，来说说进程切换的过程。进程切换的时候，会执行下面五步：（1）保存当前进程的 CPU 寄存器值（PC,R0~R13）到本进程的栈中；（2）保存 SP （用户栈栈顶）到 task->thread.save 中；（3）从新进程的 task->thread.save 中恢复 SP 为新进程的用户栈，完成栈切换；（4）从新进程的栈中恢复 CPU 相关寄存器值，完成上下文；（5）新进程开始运行，完成任务切换。

那么谁来完成进程切换？答：内核代码。当系统时钟中断到来，开始执行内核代码：检测到当前进程时间片用完，调用 schedule() 函数，内部调用 switch_mm() 和 switch_to()，前者切换页表，后者切换上下文。这时候都是运行在内核代码里面，这些函数调用链，使用的是内核堆栈，使用的虚拟地址，就是通过上面讲过的那些页分配接口、字节分配接口得到的内核虚拟地址，使用的是内核的映射关系，内核映射页表，没有使用切换和被切换的进程页表！

总结整理一下上面的要点：

  1. 进程用户栈和内核栈是分开的，不交叉；
  2. 内核代码使用内核栈，每次从用户态切换到内核态，内核栈为空；
  3. 内核态虽然是在进程上下文，但是使用的是内核映射页表，使用前面讲述的接口得到的内核虚拟地址。

现在来说说在内核栈要注意的事情。内核栈大小是固定的，不像用户栈是动态变化的，只要不冲突到动态库加载和堆，就可以一直增长。内核栈大小可配置，一页或者两页，如果是一页，那么内核的中断就有单独的中断栈；如果是两页大小，那么中断也会使用内核栈。

知道了内核栈大小是较小且固定，所以在编写内核代码的时候要注意的就是不要分配大的自动变量，否则容易越过内核栈而破坏掉别的内核数据。所以，对于那些会占用较大内存的函数，为其变量使用动态分配（alloc_pages() 等）的方式来获取内存，这样的话就是内核的页表映射问题，就不会使用可怜的内核栈。

高端内存映射

在 X86 体系结构上，高于 896M 的内存是属于高端内存，高端内存的页不会永久的（线性映射）或者自动的映射到内核地址空间。分配一个页之后，如果内核想要使用这个页，就得使用逻辑地址通过页表映射关系来操作对应的物理内存，换句话说，内核代码使用的内存操作的是其内核地址空间内的逻辑地址。

通过 alloc_pages() 获得的 struct page* 结构是没有被映射到内核的虚拟地址空间的，所以没有虚拟地址。这时候就需要函数来将一个物理页映射到内核地址空间里面，得到一个逻辑地址。

临时映射和永久映射的一个最本质区别在于，临时映射是不能睡眠的，而永久映射可以睡眠。

内核空间的内存，在高端内存区域，前面曾说过还细分成了三部分：vmalloc area、持久内核映射区、临时内核映射区。这三个区内核都为其预留了映射地址的，上面的永久映射只会被映射到该区对应的逻辑地址空间里面，同理，临时映射也是只会映射到这个区域。

每个 CPU 的分配

在 SMP 体系结构上，一般会对一些处理器数据用数组的方式来管理，每个处理器对应的数据就是在那个数组里面，以处理器编号为下标对应的数据。然后在使用这些数据的时候，先过去当前处理器编号，禁止内核抢占（避免进入相同代码修改本处理器的数据），然后从那个数组里面获取数据，使用完之后，恢复内核抢占。

新的每个 CPU 接口

编译时 percpu 数据

接口 描述
DEFINE_PER_CPU(type,name) 定义编译时 percpu 数据。该语句为系统中每个 CPU 创建一个类型为 type，名字为 name 的变量实例
DECLARE_PER_CPU(type,name) 对上面定义的 percpu 变量，在别的地方进行声明
get_cpu_var(name) 该函数返回当前处理器上名为 name 的 percpu 数据，同时禁止内核抢占
put_cpu_var(name) 重新激活内核抢占
per_cpu(name,cpu) 返回指定处理器上名为 name 的 percpu 变量，但是不会禁止内核抢占

注意：编译时的 percpu 数据不能在模块中使用。

运行时 percpu 数据

接口 描述
void* alloc_percpu(type) _alloc_percpu( sizeof(type), __alignof_\(type) )
void* __alloc_percpu(size_t size, size_t align) 第一个参数：动态创建的 percpu 变量的大小（单位字节）；第二个参数：分配时的字节对齐
void free_percpu(const void*) 释放所有处理器上的 percpu 变量
get_cpu_ptr(void* ptr) 该函数返回指向当前处理器上名为 name 的 percpu 数据的指针，同时禁止内核抢占
put_cpu_ptr(void* ptr) 重新激活内核抢占

上面的内容中需要注意一些点，同时我在阅读 linux 源码的时候，发现一些现象也一并记录于下

  1. _alignof_\ 是 gcc 的一个功能，返回一个类型或左值的对齐字节数。
  2. 对于使用 DEFINE_PER_CPU(type,name) 而定义的 percpu 数据，大多使用 per_cpu(name,cpu) 来操作；
  3. 对于运行时的 percpu 数据，大多使用 get_cpu_ptr() 来操作。

为啥要使用 percpu 数据？

  1. 减少数据锁定，提升系统性能 。只要能保证那些 percpu 处理器数据只能被一个处理器访问，那么就不需要锁，只需要禁止内核抢占就行了。
  2. 大大减少缓存失效。 缓存失效发生在处理器试图使他们的缓存保持同步时。如果一个处理器操作的数据存放在别的处理器的缓存上，那么当这个处理器要处理这个数据的时候，现在保存这个数据的处理器要么清除要么刷新自己的缓存，当时如果保证了数据只出现在某个处理器的缓存上，那么就不会有这种缓存失效的问题。（持续不断的缓存失效被称为缓存抖动）
