原子操作

所谓原子操作指的是不能被分割的原子指令。内核提供了两组原子操作的接口，一组针对整数，一组针对位。在 linux 所支持的所有体系结构上，都实现了这两组接口。

在使用原子操作接口的时候，引入了一个新的数据类型，原子数据类型，atomic_t，而没有直接使用 C 语言的 int 类型，这是因为：

  1. 让原子函数只接受 atomic_t 类型的参数，可以保证原子操作只与这种类型的数据一起使用；
  2. 保证该类型的数据不会被传递给非原子函数；
  3. 使用 atomic_t 类型的数据可以确保编译器不对其做访问优化，使得每次取得的值总是该原子类型数据在内存上的值；
  4. 在不同体系结构上实现原子操作的时候，可以通过这中间层屏蔽底层实现的差异。

原子性确保指令执行期间，要么全执行完，要么根本不执行。

在编写代码的时候，能使用原子操作的时候，就不要使用复杂的加锁机制，因为对多数体系结构来说，原子操作相较于更复杂的同步方法来说，给系统带来的开销更小，对高速缓存行的影响也较小。

自旋锁 spin lock

自旋锁最多只能被一个可执行线程拥有，如果当一个线程已经拥有了一个自旋锁，另一个线程想要获取这个自旋锁，那么就会一直循环，忙等待别的地方对这个自旋锁解锁。这里的忙等待就是一直在 while 探测，所以被称为自旋。如果锁没有被获取，那么请求锁的线程会立即得到它，并且继续执行。

一个被争用的自旋锁使得请求它的执行线程在等待锁重新可用时自旋，这种行为是自旋锁的特点。自旋的时候特别浪费处理器时间，所以自旋锁的持有时间应该尽可能短。自旋锁的使用目的就是：能够短时间内轻量级加锁。

自旋锁的使用

因为自旋锁在同一时刻只能有一个线程可以持有，所以一个时刻只可能有一个线程位于临界区中，这就为多处理器机器提供了防止并发访问的保护机制。

警告！！！linux 实现的自旋锁是不可递归的。因为递归获取锁的时候是第二次获取锁，这时候锁已经被持有，就得等待自旋锁的解锁，但是解锁需要返回到上一层，所以就造成了上一层没有办法解锁，当前层没有办法获取锁的死锁状态。

自旋锁可以在中断处理程序中使用，但是在获取一个自旋锁之前， 应当先关闭本地处理器的中断，否则，如果本地处理器的中断重新触发，使得又进入了这个中断处理函数（不同的中断线注册了同样的中断处理函数），那么就会造成双重请求，这次的中断因为自旋退不出来，上层的中断因为这次中断没有结束没办法继续执行以解锁，就造成了死锁。关闭的中断只需要本处理器的中断就行了，因为即使另一个处理器也执行该中断处理函数要获取自旋锁，也不会影响这个处理器的执行，所以这个处理器的自旋锁会释放，另一个处理器的代码就可以得到锁并继续执行。

自旋锁与下半部

由于下半部可以抢占处于进程上下文的代码，所以当下半部与进程上下文有共享数据的时候，在获取进程中的共享数据时，不仅要加锁，还要禁止下半部。因为用户进程是不会抢占下半部执行的，所以只用考虑一个方向。函数 spin_lock_bh() 用于获取指定的锁，同时会禁止所有下半部的执行；spin_unlock_bh() 是激活下半部并解锁。

上述的道理适用于中断和下半部。中断可以抢占下半部的执行，所以如果中断与下半部共享数据，那么下半部在获取共享数据的时候，应该禁止中断，同样，因为下半部不会抢占中断，所以只考虑一个方向。

对于 tasklet 而言，不同种类的 tasklet 可以在不同处理器上并行执行，所以也会涉及到加锁问题，不过这里使用简单的自旋锁就行了，因为只是并行访问的问题，没有抢占，而且同一个 tasklet 是不能在不同处理器上同时执行，所以不用禁用下半部，因为即使另一个 tasklet 在自旋等待共享数据，这边的 tasklet 一定会释放锁的。

对于软中断而言，只要是共享变量，只要是全局变量，都要加锁，因为同一个软中断可能在不同的处理器上同时执行，这个与 tasklet 不同。不过因为一个处理器上有软中断在执行的时候，会禁止该处理器的软中断执行，所以也不用禁止下半部，因为不会被本处理器所抢占，而自己总会释放锁，不会导致另一个处理器相同软中断的死锁。

读-写自旋锁

当对某个数据结构的操作可以明确的划分出 “读/写” 或者 “消费者/生产者” 两种类别的时候，就可以使用这种读写自旋锁。

读写自旋锁的规则如下：

  1. 一个或者多个读任务可以并发的持有读者锁；
  2. 写锁只能由一个写任务持有；
  3. 如果当前共享变量有被持有的读锁，那么写任务需要自旋等待所有读锁释放之后才能持有写锁；
  4. 当共享变量有被持有的写锁时，读任务需要自旋等待写任务释放写锁之后才能持有读锁。

信号量

自旋锁的初衷，是在短时间内轻量级加锁。但是如果加锁时间变长且代码可能会睡眠，因为另一方是自旋等待，所以会浪费大量的处理器时间，所以对于这种情况下，就应该选择睡眠锁。

信号量是一种睡眠锁。如果一个任务试图获取一个已经被占有的信号量，那么该任务会被推到所请求信号量对应的一个等待队列，然后修改其线程状态为挂起态（TASK_INTERRUPTIBLE or TASK_UNINTERRUPTIBLE），让其睡眠，重新调度别的程序执行，所以信号量被称为睡眠锁。

由上面的描述可以知道，信号量的开销，至少是有两次上下文切换，一次是睡眠切出，一次是被唤醒的切入。而自旋锁，其开销就是在等待上面。

可以从信号量的睡眠特性上得出一些结论：

  1. 争用信号量的进程在等待信号量的时候会睡眠，所以信号量适用于锁会被长时间持有的情况。
  2. 如果锁被短时间持有，那么就不适合使用信号量，因为睡眠、维护等待队列以及唤醒，所花费的开销可能比锁被占有的时间还要长。
  3. 由于执行线程在争用锁的时候会睡眠，所以信号量只能在进程上下文中才能获得，不能使用于中断上下文，因为中断上下文是不能被调度的。
  4. 可以在持有信号量之后睡眠，因为别的争用信号量的进程得不到信号量也只是睡眠等待而已，不会造成死锁。
  5. 在占用信号量的同时，不能占用自旋锁。因为持有信号量的时候也可能会睡眠，但是这时候又持有自旋锁，而自旋锁是不允许睡眠的。

至于何时使用自旋锁，何时使用信号量，首先判断当前获取锁的进程是否会睡眠，如果会。那么也只能选择信号量；如果不会，那么就要根据持有锁的时间长短来作为主要参考，持有时间短，自旋锁，持有时间长，信号量。

信号量不同于自旋锁，他不会禁止内核抢占，所以持有信号量的代码是可以抢占的，被抢占的情况就相当于他主动睡眠了，其余的争用信号量的进程也是同他一起在睡眠而已。

计数信号量与二值信号量

信号量可以同时允许任意数量的锁持有者，而自旋锁在一个时刻只允许有一个持有者。信号量允许的持有者数量被称为使用者数量，简称为数量。信号的使用者数量值为 1 的时候，这就是一个二值信号量，或互斥信号量；如果大于 1 ，就是计数信号量。

信号量支持两个原子操作，P() 和 V()，这里的 pv 是来自荷兰语 Proberen 以及 Vershogen，前者表示测试，后者表示增加。在 linux 系统里面，他们对应的接口是 down() 以及 up() 。down() 操作通过对信号量减一来请求获取一个信号量，如果减一之后的结果是 0 或者大于0，获取该信号量，任务进入临界区。如果结果为负数，则该任务会被放入到等待队列，处理器执行其他任务（也就是这里会执行一次调度）。当临界区的操作完成之后，up() 操作用来释放信号量，会将信号量的数值增加一。up()操作会检车该信号量的等待队列是否为空，如果不为空，那么处于等待队列中的任务在被唤醒的时候会获得该信号量。

读写信号量

读写信号量和信号量之间的关系，与自旋锁和读写自旋锁之间的关系类似。读写信号量在内核中，是由 rw_semaphore 结构体表示。

所有的读写信号量都是互斥信号量，只对写者互斥。并发持有读锁的读者数量不限，但也是只有唯一的写者可以获取写锁。所有的读写信号量都不会被信号打断。

读写信号量也提供了 down_read_trylock() 和 down_write_trylock() 方法。 呵呵 ，呵呵，呵呵，他们都是锁住了返回非 0，没锁住返回 0。与信号量自己的都是反的，呵呵呵呵。

互斥体 MUTEX

互斥体也是一种睡眠锁，相较于信号量而言是一种更简单的强制互斥睡眠锁，而且互斥体可以自动调试。

mutex 的行为和使用计数为 1 的信号量行为类似，但是实现更高效，而且使用也有更强的限制。

mutex 的基本操作列表

方法 描述
mutex_lock(struct mutex *) 为指定的 mutex 上锁，如果锁不可用则睡眠
mutex_unlock(struct mutex *) 为指定的 mutex 解锁
mutex_trylock(struct mutex *) 尝试获取锁，成功锁住返回 1，否则返回 0
mutex_is_locked(struct mutex *) 如果锁被争用返回 1，否则返回 0

mutex 的使用场景更严格：

  1. 任何时刻，只能有一个任务可以持有 mutex 锁；
  2. 给 mutex 上锁者必须为其解锁！ 不能在一个上下文中锁定一个 mutex，然后在另一个上下文中为其解锁。但是信号量和自旋锁可以。
  3. 递归上锁和解锁是不允许的。不能对已经加锁的 mutex 再加锁，同样也不能对一个已经解锁的 mutex 再解锁。
  4. 当持有一个 mutex 的时候，进程不可以退出（是不能被调度的意思吗？）。
  5. mutex 不能在中断或者下半部中使用，同样，也不能使用 mutex_trylock();
  6. mutex 只能通过官方 API 进行管理，不能拷贝、手动初始化或者重复初始化。

信号量和互斥体

除非上述 mutex 的某个约束条件妨碍你的代码实现，否则相比于信号量，优先使用 mutex。其实一般只有靠近底层的时候才会需要使用信号量。

自旋锁和互斥体

在中断上下文中，只能使用自旋锁，在任务睡眠的时候只能使用互斥体（信号量）。

加锁选择的建议：

需求 建议加锁方法
低开销加锁 优先使用自旋锁
短期加锁 优先使用自旋锁
长期加锁 优先使用互斥体
中断上下文中加锁 使用自旋锁
持有锁需要睡眠 使用互斥体

完成变量

好像和系统接口的条件变量有点相似。

如果在内核中一个任务需要发出信号通知另一个任务发生了某个特定事件，利用 完成变量 可以做到。

等待任务调用 wait_for_completion(&my_comp) 在指定的完成变量上等待事件发生；而另一个任务当完成某个事件的时候调用 complete(&my_comp) 来给等待的任务发送信号以唤醒继续执行。

方法 描述
init_completion(struct completion *) 初始化指定的动态创建的完成变量
wait_for_completion(struct completion *) 等待指定的完成变量接收信号
complete(struct completion*) 发送信号唤醒任何等待的任务

顺序锁

简称 seq 锁，是在 2.6 版本中才引入的新型锁。

实现这种锁主要依靠一个序列计数器。 序列计数器的初值是 0，读锁只会读取序列计数器值，不会修改；而写锁是递增改变序列计数器的值：当获得写锁的时候，序列计数器值加一，此时在写者临界区中，序列计数器值为奇数；当写者完成工作之后释放写锁，序列计数器值再加一，又变成了偶数。而读者在读之前和读之后都会读取序列计数器值并检查。

  1. 如果前后两次读取的序列计数器值相同，则表示读取过程中没有被写者修改过；
  2. 如果在读之前读取的序列计数器值是奇数，表明当前正有写者在修改数据，读者不能读取数据（如下例程中，会一直循环）；
  3. 如果读之前是偶数，读完之后发现这次读取到的序列计数器值与读取数据之前的序列计数器值不同，那么就表明期间该数据被写者修改了（也会导致 while 判断条件为真）；

禁止抢占

对于每个处理器特有数据，即只有这个处理器才会去访问，那么其实可以不用锁，但是却会出现在该处理器上某个进程再处理这个数据的时候，执行了调度，发生了内核抢占，新的被调度的程序也会处理这个数据，就造成了数据的破坏。

为了解决上面的问题，在不加锁的情况下，可以通过禁止内核抢占来达到对处理器特有数据的同步效果。

之前在讲调度的时候，说到过：

为了支持内核抢占，在 thread_info 结构中添加了一个 preempt_count 计数器，初值为 0 ，使用锁，计数器加 1 ，释放锁，计数器减 1 。当 preempt_count 计数器为 0 ，内核可抢占执行。

下面接口中所说的计数值就是指的该变量的值：

函数 描述
preempt_disable() 增加抢占计数值，从而禁止该内核抢占
preempt_enable() 减少抢占计数值，并当该值降为 0 时检查和执行被挂起的需要调度的任务
preempt_enable_no_resched() 激活内核抢占但不再检查任何被挂起的需要调度的任务
preempt_count() 返回抢占计数值

preempt_disable() 和 preempt_enable() 是可以嵌套调用的，当 preempt_disable)() 调用了 N 次，那么 preempt_enable() 也要调用 N 次才是真正的开启了内核抢占。

禁止内核抢占相关接口是有效的，对原子操作以及睡眠的调试方法。

顺序和屏障

当处理多处理器之间或者硬件设备之间的同步问题时，有时需要在你的程序代码中以指定的顺序发出读内存、写内存指令。有时候的需求是按照写的顺序来读数据。但是编译器和处理器为了优化，可能会对没有明显关联的读与写进行重新排序。拿处理器来说，因为使用的是流水线技术，所以要是相邻语句有同时使用共同部件的情况，那么就可能从后面的语句中，挑出与这两条语句不相干的，然后插入到这两条语句中间执行，就能提高处理器的吞吐率。

但是上述的优化操作，会在确实存在依赖关系，但是依赖关系不明显的多个变量间，产生数据操作与预期不符的情况。这时候就需要使用 屏障（barriers） 来保证数据的操作顺序是按照预期顺序在执行。

rmb() 方法提供读内存屏障（Read Memory Barriers），确保跨越 rmb() 这条语句的读取动作不会发生重排。换句话说，在 rmb() 之前的读取动作不会被重排到 rmb() 语句之后，同样，在 rmb() 之后的读语句，也不会被重排到 rmb() 之前。

wmb() 提供的是写内存屏障，与 rmb() 使用方法一样，不过就是针对的是写操作的重排。

mb() 提供的就是读写屏障。read_barrier_depends() 提供的是依赖数据之间的读屏障。

barrier() 方法可以防止编译器跨屏障对载入或者存储操作进行优化。他只防止编译器可能重排指令

屏障 描述
rmb() 阻止跨越屏障的载入动作发生重排序
read_barrier_depends() 阻止跨越屏障的具有数据依赖关系的载入动作重排序
wmb() 阻止快约屏障的存储动作发生重排序
mb() 阻止跨越屏障的载入和存储动作重新排序
smp_rmb() 在 SMP 上提供 rmb() 功能，在 UP 上提供 barrier() 功能
smp_read_barrier_depends() 在 SMP 上提供 read_barrier_depends() 功能，在 UP 上提供 barrier() 功能
smp_wmb() 在 SMP 上提供 wmb() 功能，在 UP 上提供 barrier() 功能
smp_mb() 在 SMP 上提供 mb() 功能，在 UP 上提供 barrier() 功能
barrier() 阻止编译器跨屏障对载入或存储操作进行优化
