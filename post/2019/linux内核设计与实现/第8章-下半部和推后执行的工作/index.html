从第七章的学习知道了中断处理程序的特点：

  1. 以异步方式执行，会打断其他重要代码，所以为了避免被打断代码停留时间过长，所以中断处理函数的执行需要快速；
  2. 由于处于中断上下文，所以不可阻塞；
  3. 会关闭中断系统（设置 IRQF_DISABLED 则屏蔽所有处理器接收任何中断信号的能力，没有设置 IRQF_DISABLED 则对所有处理器屏蔽当前中断所在中断线）；
  4. 由于中断处理程序往往需要对硬件进行操作，所以他们通常具有更高的时限要求。

所以中断处理程序在整个硬件中断处理流程中仅占一部分的原因就比较明显了。操作系统必须有一个异步，快速，简单的机制来负责对硬件做出迅速的响应，并完成那些时间要求严格的操作，即对硬件中断做出即时响应，这就是上半部。而把那些其他的，以及对时间要求宽松的操作，就推后到中断系统被激活之后进行，这就是下半部。

下半部

下半部执行的工作就是与中断密切相关，但是又不是中断处理函数所做的事情。对于中断处理程序而言（上半部），几乎都要通过操作硬件来对中断的到达进行确认，有时还会从硬件拷贝数据。然后剩下的工作就交给了下半部来完成。

但是并没有明确的规定，究竟什么该放在上半部执行，什么该放在下半部执行。不过有以下几点可以提供参考：

  1. 如果一个任务对时间非常敏感，那么把他放在上半部中断处理程序中执行；
  2. 如果一个任务和硬件有关，那么放在上半部；
  3. 如果一个任务要保证不被别的中断打断（特别是同一个中断线的中断），那么把他放在上半部；
  4. 其他所有任务考虑放在下半部。

为什么要使用下半部

主要的原因还是因为屏蔽中断信号。前面已经多次提及，当一个中断处理程序执行的时候，位于中断上下文，如果没有设置 IRQF_DISABLED 则会对所有处理器都屏蔽掉当前中断线，如果设置了 IRQF_DISABLED 则情况更糟，会屏蔽掉所有处理器上的所有中断信号。综上所述，当一个中断处理程序在执行的时候，总会屏蔽掉一些中断信号，而中断信号又是硬件与系统通信的唯一方式，所以中断处理程序应该尽快退出，恢复处理器接收中断信号的能力，也就是恢复了与硬件通信的能力，这种恢复，速度越快，系统的对硬件的响应性能越好。

所以对硬件中断信号的响应分为了上下部，下半部在执行的时候，是恢复了处理器接收中断信号的能力的。

在 linux 里面，把这种将一部分工作推后的机制都称为 “下半部机制”。目前，有三种方法来实现将工作推后执行：

  1. 软中断（使用较少）
  2. tasklet（基于软中断实现）
  3. 工作队列

软中断

软中断使用较少，但是 tasklet 是基于软中断实现的，所以先讲述软中断，软中断代码位于 kernel/softirq.c

软中断是在编译期间静态分配的，他不像 tasklet 那样可以动态注册注销。一个软中断由 softirq_action 结构体表示。而软中断向量 softirq_vec 数组有 NR_SOFTIRQS 个 softirq_action 项。每个被注册的软中断都占据该数组的一项。NR_SOFTIRQS 是个枚举值，位于枚举结构体最后，用以获取当前注册的软中断数量。

一个软中断不会抢占另一个软中断，他们可以同时在不同的处理器上面运行。其实唯一可以抢占软中断就只有中断处理程序。

软中断的执行过程

如前面所说，软中断是静态分配的，在 linux/interrupt.h 中，分配了如下软中断：

软中断枚举符号 相对优先级 软中断描述
HI_SOFTIRQ 0 优先级高的 tasklets
TIMER_SOFTIRQ 1 系统定时器的下半部
NET_TX_SOFTIRQ 2 发送网络数据包
NET_RX_SOFTIRQ 3 接收网络数据包
BLOCK_SOFTIRQ 4 BLOCK 装置
IRQ_POLL_SOFTIRQ 5
TASKLET_SOFTIRQ 6 正常优先级的 tasklets
SCHED_SOFTIRQ 7 调度程序
HRTIMER_SOFTIRQ 8 高分辨率定时器
RCU_SOFTIRQ 0 RCU 锁定

最后的 NR_SOFTIRQS 也是前面所提到，用于声明 softirq_vec 数组长度。

当你要往系统中添加一个软中断，

  1. 首先应该在这个枚举结构体里面声明你的软中断。

  2. 然后编写类型为 void softirq_handler(struct softirq_action *) 的软中断处理函数。

  3. 调用函数 open_softirq(NET_TX_SOFTIRQ,net_tx_action); 来注册你自己的软中断，绑定软中断处理函数，这里是用发送网络数据包的软中断注册来举例。

  4. （可以先看第五条再来看这条）在想要后面执行软中断处理函数的地方，用函数 raise_softirq(NET_TX_SOFTIRQ); 来挂起软中断信号，此后合适的时机软中断处理函数会被调用执行。

  5. 软中断在被挂起之后，会在以下三个场景被检查调用软中断处理函数：

    • 从一个硬件代码中断处返回
    • 在 ksoftirqd 内核线程中
    • 在那些显式检查和执行待处理软中断的代码中

清楚了整个流程之后，可以知道，软中断处理函数执行的时候，是可以响应中断的，并且可以对软中断位图进行设置，也正是因为可以对软中断位图进行设置，所以 do_softirq() 是不能休眠的，否则就会漏执行软中断处理函数。

还有一些应该值得注意的点：当一个软中断处理程序在执行的时候，当前处理器上的软中断被禁止，但是其他处理器仍然可以执行 do_softirq()，所以对于软中断处理程序内部使用的静态变量，或者是共享数据，都应该要有锁机来保证同步。

在中断处理函数中触发软中断是最常见的形式，中断处理函数中将要执行的软中断挂起，然后退出中断处理函数。从中断处理函数返回之后会立马执行 do_softirq() ，就开始执行中断的下半部。

tasklet

tasklet 是利用软中断实现的一种下半部机制，他与进程没有任何关系。tasklet 的接口更简单，锁保护要求也更低。

tasklet 的实现

因为 tasklet 是利用软中断实现的，所以其实他们本身也就是软中断。如前面所说，有两个软中断是跟 tasklet 相关，HI_SOFTIRQ 和 TASKLET_SOFTIRQ，区别只在于相对优先级。

tasklet 由 tasklet_struct 结构体表示，一个 tasklet_struct 结构体对象就是一个单独的 tasklet。

func 就是 tasklet 的处理程序，就相当于是软中断中的 action，data 成员是 func 的唯一参数。

state 成员只能在 0、TASKLET_STATE_SCHED、TASKLET_STATE_RUN 之间取值。TASKLET_STATE_SCHED 状态相当于软中断执行了 raise_softirq() 之后的状态，等待被执行。而 TASKLET_STATE_RUN 状态就是其字面意思，正在运行中，别的处理器再检查 tasklet 状态的时候，如果是正在运行，那么就跳过该 tasklet 转而执行下一个，所以同一个 tasklet 不能在不同的处理器上同时运行，原因就在这里，这也是 tasklet 比软中断对锁的要求低的原因，因为同一个 tasklet 是不能同时执行的。

count 成员是 tasklet 的引用计数器。如果不为零，则表示 tasklet 被禁止，不允许执行；如果为 0 ，tasklet 才是被激活的，在激活状态下，如果还处于 TASKLET_STATE_SCHED 状态，那么这个 tasklet 就可以在合适的时候执行处理函数 func。

设置 tasklet 为调度态

首先一点，上述的 tasklet_schedule() 和 tasklet_hi_schedule() 都是会在中断处理函数里面被调用，而这两个函数的作用就是把对应的 tasklet_struct 结构类型的参数加入到相关的 tasklet 链表队列里面，并且挂起软中断，等中断处理程序退出之后，就会执行软中断对应的处理函数，而 tasklet 软中断的处理函数就是检查 tasklet 链表中的每一个 tasklet_struct 对象，是否可执行，可以的话就执行，否则跳到下一个继续检查执行。

tasklet_schedule() 的具体流程如下：

  1. 检查参数指定的 tasklet_struct 是否已经是被调度了，如果是，直接退出函数。这种情况一般出现在前面该 tasklet 已经执行了一次 tasklet_schedule() 函数，但是还没有来得及执行，就又调用了一次 tasklet_schedule()。
  2. 进入 __tasklet_schedule，保存中断状态，并且禁用本地中断。这样做是为了在处理 tasklet_schedule 的时候，没有新的中断产生与之相关的设置与请求，以保证处理器状态的有效。
  3. 把需要被调度的 tasklet ，即参数指定的 tasklet 加入到对应的链表中。
  4. 挂起 TASKLET_SOFTIRQ 信号，以便中断返回的时候执行的 do_softirq() 。
  5. 恢复中断状态，返回。

上面的 tasklet_schedule 可以是 tasklet_hi_schedule，TASKLET_SOFTIRQ 软中断相应的就是该替换为 HI_SOFTIRQ 软中断。

TASKLET_SOFTIRQ（HI_SOFTIRQ） 的软中断处理函数 tasklet_action() (tasklet_hi_action()) 具体流程如下：

  1. 禁止中断，为当前处理器检查 tasklet_vec 链表。
  2. 将链表从 tasklet_vec 表头上摘取下来，然后将 tasklet_vec 置为 NULL 以清空 tasklet_vec 链表。
  3. 打开中断。
  4. 循环遍历摘取下来的链表上的每一个 tasklet_struct。
  5. 如果是多处理器，通过检查 state 成员状态是否是 TASKLET_STATE_RUN，来判断当前的 tasklet 是否在别的处理器上被执行，如果是则跳到下一个 tasklet。
  6. 如果当前 tasklet 没有执行，那么设置其状态为 TASKLET_STATE_RUN。
  7. 检查 count 成员是否为 0，确保该 tasklet 没有被禁止，如果当前 tasklet 被禁止，那么就跳到下一个 tasklet 执行。
  8. 确定当前状态不是 TASKLET_STATE_RUN，count 是 0，执行该 tasklet 绑定注册的处理函数。
  9. tasklet 执行完成之后，将 state 清零。
  10. 继续下一个 tasklet，直至没有待处理 tasklet。

第一步和第三步关于中断的操作，因为这个程序一定是运行的在软中断的处理函数里面，所以在这个状态下，处理器是应该接收所有的中断的，也就是在这个状态下，禁止中断之前，处理器也是会处理所有中断，故而这里就直接是禁用全部中断以及打开全部中断，而不是保存中断状态再恢复。

使用 tasklet

  1. 编写 tasklet 处理函数。

    有关 tasklet 处理函数的注意事项：

    • 因为 tasklet 是用软中断实现的，所以 tasklet 不能睡眠，也就意味着不能在 tasklet 里面使用信号量或其他可能导致阻塞的函数。
    • tasklet 运行时，允许中断，所以如果有共享数据，应该注意加锁保护。

  2. 声明自己的 tasklet 。
    可以静态创建，也可以动态创建。对于静态创建而言，有两个宏可以使用：DECLARE_TASKLET(name,func,data) 和 DECLARE_TASKLET_DISABLED(name,func,data)，这两个宏的区别在于对成员 count 的初值设置不同，其中 disabled 设置的 count 初值为 1，是禁止状态。

  3. 调度这个自己的 tasklet 。
    调度方式就是使用前面一小节所提到的 tasklet_schedule()，在中断处理函数中使用这个函数。

此外还有一些跟 tasklet 相关的函数：

函数名 说明
tasklet_disable(&my_tasklet); 通过将 count 原子变量递增 ++ 来禁止 my_tasklet
tasklet_enable(&my_tasklet); 通过将 count 原子变量递减 – 来试图允许 my_tasklet(这里说试图是因为判断一个 tasklet 是否被禁止是判断 count 是否等于 0)
tasklet_kill(&my_tasklet); 从挂起的队列中去掉这个 tasklet。不过不影响其他地方重新调度该 tasklet

ksoftirqd 内核线程

因为 tasklet 是通过软中断来实现，所以下面讨论的关于软中断，同样适用于 tasklet。

往往，软中断都是在中断处理函数中被挂起，然后中断处理函数返回之后，软中断开始检测执行。而且，有的软中断还会挂起其本身，所以系统中是有可能一直存在软中断待执行的。

当系统中存在了大量的软中断，如果一直在处理他们，那么用户态程序就可能处于饥饿状态；如果响应第一次软中断的挂起，而对自己再次挂起的软中断留待第二次中断到来的时候再处理，那么对于一个空闲的系统来说，自己再次挂起的软中断会处于饥饿状态。

为了平衡上述的两种情况，内核有一个线程 ksoftirqd 来负责辅助处理。对于重新触发的软中断，系统不会立即处理，并且当系统中有大量软中断出现的时候，内核会唤醒 ksoftirqd 来处理这些负载。ksoftirqd 的 nice 值是 19，是一个低优先级的线程，以此来避免他跟重要任务争夺资源。这样，软中断就一定能够得到处理，并且用户程序也不会处于饥饿状态。

每个处理器都有一个这样的内核线程，名为 ksoftirqd/n，其中的 n 在多处理器上就是对不同处理器的编号。一旦该线程被初始化，他就会执行类似下面的死循环：

工作队列

工作队列可以把任务推后，并且交给一个内核线程来执行。这个下半部总是运行在进程上下文中，这样，通过工作队列来实现的下半部就具有进程上下文的优势，可以睡眠并且允许被重新调度。

在工作队列和 tasklet 中选择使用哪个是比较容易的，如果任务需要睡眠或者重新被调度，那就选择工作队列；如果推后执行的工作不需要睡眠，那么可以选择软中断或者是 tasklet。

工作队列是唯一一个可以运行在进程上下文的下半部机制，也只有他在下半部机制中是可以睡眠的。

下半部机制的选择

  1. 如果有睡眠需求，如果要是可调度实体，那么只能选择工作队列；
  2. 如果没有上述的限制，并且中断次数发生较多（导致多次执行下半部），这时候虽然可以使用工作队列，但是工作队列有进程间切换的开销，相比之下，tasklet 会是一个更好的选择；
  3. 软中断是最麻烦的一种下半部机制，需要在内核源码里面静态添加软中断，但是如果追求性能的提升，也是可以考虑软中断来实现下半部。
下半部 上下文 顺序执行保障
软中断 中断上下文 没有
tasklet 中断上下文 同类型不能同时执行
工作队列 进程上下文 没有，是通过调度器调度

在下半部之间加锁

对于软中断和工作队列，他们的执行顺序是不定的，有可能上一次中断退出之后执行了一个下半部，在执行过程中，相同的中断发生，会导致后面又执行相同的下半部，这时候就会有一些同步问题发生。相对而言，在这点上，tasklet 因为不允许相同的 tasklet 同时执行（即使是不同的处理器），所以也就不用担心这个问题。

然而对于三者，都要关系的就是下半部之间的同步问题，对于一些共享变量，仍然需要合适的加锁机制。

  1. 如果是 进程上下文 和 下半部 共享数据，在访问数据之前，需要先禁止下半部的处理并得到锁的使用权。这么做是为了本地和 SMP 的保护并且防止死锁的出现。（这里是谁在访问数据之前要禁止下半部书上没说，我暂时还不清楚，只是做个笔记）
  2. 如果是 中断上下文 和 下半部 共享数据，在访问数据之前，需要先禁止中断并得到锁的使用权。这么做是为了本地和 SMP 的保护并且防止死锁的出现。
  3. 任何在工作队列中共享的数据也需要使用锁机制，需要注意的点和一般内核代码相同。

禁用下半部

为了保证共享数据安全，一般的做法是先得到一个锁，然后再禁用下半部的处理。被禁用的下半部是所有的软中断以及所有的 tasklet。为什么只禁止软中断和 tasklet 而不禁止工作队列？因为软中断的 tasklet 的执行是异步发生的，他们的执行是在中断返回之后，也就是可能会打断当前正在执行的被保护临界区。而工作队列是通过调度执行，运行在进程上下文，所以不存在这个问题。

函数 描述
void local_bh_disable() 禁止本地处理器的软中断以及 tasklet 的处理
void local_bh_enable() 激活本地处理器的软中断以及 tasklet 的处理

有一个值得注意的地方，他们可以嵌套调用，多次禁止仍然是保持禁止状态，而激活的次数需要与禁止的次数相同，才会真正激活软中断以及 tasklet。
