中断

中断的本质上是一种特殊的电信号，由硬件发送给处理器。这种发送的过程就是对应引脚的电平变化。 处理器检测到这个信号之后，就中断自己当前的工作转而处理中断。这里的处理中断就是操作系统提供的一段代码，从操作系统的角度来看，是从处理器获取中断到来的信息；从处理器的角度来看，就是通过这段处理中断的代码来告知操作系统，有中断到来了。此后就是操作系统来处理中断了。

不同设备对应的中断不相同，每一个中断都通过一个唯一的数字来标识，这样操作系统就能区分到底是产生了什么中断。这种唯一的数字，也被称为中断请求线，不过我还是习惯称其为中断请求号，IRQn。 比如在经典 PC 上，IRQ0 是时钟中断，IRQ1 是键盘中断。不过在 PCI 总线上的设备，中断号是动态分配的。不过只要记住，一个中断对应一个唯一的中断号。

中断的产生是异步的，硬件在产生中断的时候，是不会考虑处理器的时钟周期的，随意中断可能在任意时刻产生。而异常则是同步由处理器产生。

中断处理程序

上面说过，硬件通过电信号中断了处理器的执行之后，处理器得知有硬件产生了中断，会把这个中断消息告知操作系统，然后由操作系统接手这个中断的处理。为了响应收到的中断，内核会执行与该中断对应的函数，这个函数就被叫做中断处理函数，或者是中断服务例程（ISR Interrupt Service Routine)。

一个设备会产生多少个中断，就有多少个中断处理函数与这个设备对应。一个设备的中断处理函数是这个设备的驱动程序的一部分，设备的驱动程序是用于对设备进行管理的内核代码。

在 linux 里面，中断处理函数就是普通的 C 函数，只不过要按照特定的类型声明，以便内核可以按照标准的方式传递处理程序的信息，其他方面都和一般函数没有区别。中断处理程序与一般函数的最本质区别在于，中断处理程序被内核调用，用以响应中断，并且运行在中断上下文中，在中断上下文中执行的代码不能被阻塞。

上半部与下半部

中断可能随时发生，所以中断处理程序可能随时被执行。为了能够尽快恢复被中断代码的执行，中断处理程序应该快速的执行，或者说，中断处理程序应该执行轻量的任务，然后快速退出。从后面的中断处理流程中可以看到，当收到一个中断之后，处理器会停止正在做的事情，跳转到特定位置开始执行中断处理函数，并且关闭中断系统。这里的关闭中断系统，如果被调用的中断处理程序设置了 IRQF_DISABLED，那么会关闭所有的中断线，即处理器在中断函数执行期间，不接受任何新的中断；而如果没有设置 IRQF_DISABLED，那么会屏蔽掉当前中断线。 所以这也要求中断处理分出上下部。

如果考虑一下类似网卡这样的设备产生的中断，有大量的数据是需要拷贝和处理，如果把所有操作都放到中断处理程序中，那么这个中断处理程序显然不是轻量的，所以 为了能够让中断处理程序运行的又快，完成的工作量又多，就把中断的处理切分成了两个部分，上半部和下半部。

中断处理程序的上半部： 接收中断之后立即开始执行，但是只做有严格时限的工作，例如对接收到的中断进行答复，或者复位硬件。这些工作都是在所有中断被禁止的情况下完成的。

中断处理程序的下半部： 这部分完成那些允许被延后执行的工作。在之后的合适时机，下半部会被开中断执行。具体内容见 第八章。

注册和卸载中断处理函数

如果一个设备要使用中断，那么其驱动程序中就应该有注册中断处理程序的代码。

注册中断处理程序

驱动程序可以通过 request_irq() 函数注册一个中断处理程序，并且激活指定的中断号，用于之后处理对应的中断。

  1. unsigned int irq ：
    irq 表示要分配的中断号。对于传统的 PC 设备，如键盘，鼠标等，是可以预先确定，不过对别大多数别的设备而言，可以通过探测获取，或者是通过编程动态获取。（怎么探测，怎么动态获取我还不知道。）

  2. irq_handler_t handler ：
    handler 是一个函数指针，指向处理这个中断的中断处理函数，只要操作系统一接收到这个中断，handler 所代表的实参这个函数就会被内核调用。
    typedef irqreturn_t (*irq_handler_t)(int,void*);

  3. unsigned long flags ：
    flag 可以为 0 ，也可以是下面的一个或者多个标志的位掩码，标志中比较重要的有：

    • IRQF_DISABLED：如果该标志被设置，表示内核在处理中断处理程序的时候，所有别的中断都将被禁用。如果不设置，该中断线可以与其他中断线同时运行。
    • IRQF_SAMPLE_RANDOM：如果有中断设置了这个标志，那么中断的间隔时间就会被添加进内核熵池，此后可以作为硬件随机值被提供。
    • IRQF_TIMER ：该标志是为系统定时器专门准备的。
    • IRQF_SHARED ：表明这个中断号以及中断处理程序会被多个设备共享，区分对应于同一个中断号的不同设备，是通过 request_irq() 的最后一个参数。要是注册中断处理程序的两个不同设备使用了同一个中断号，但是没有使用 IRQF_SHARED 标志，那么后注册的注册函数会注册失败，返回 -EBUSY。

  4. const char* name ：
    与中断相关的设备的 ASCII 文本表示，这个值会被 /proc/irq 和 /proc/interrupts 文件所使用。

  5. void* dev ：
    如果这个参数有唯一确定性，那么就可以作为一个 cookie ，用来区分共享了中断号的不同中断处理程序、或是共享了中断处理程序的不同设备。另外，dev 也可能指向中断处理程序使用的一个数据结构，因为对每个设备而言，设备结构都是唯一的，而且可能在中断处理程序中会使用到这个结构。并且，如果要卸载一个中断处理程序的时候，这个 dev 也能提供一个唯一标识信息。

request_irq() 成功执行则返回 0 ，否则返回非 0，指定的中断处理程序就不会被注册。

注意！request_irq() 函数可能会休眠，所以不能在中断上下文或者其他不允许阻塞的代码中调用该函数。 request_irq() 会休眠的原因是因为他会在 /proc/irq 文件中创建一个对应项，所以层层调用下，会调用到 kmalloc() 函数，而这个函数是可以休眠的。

另外还有一点需要注意的是，要在设备初始化之后再注册中断函数，避免设备未初始化完成就在执行中断处理函数。

卸载中断处理程序

  1. 如果指定的中断号不是共享的，那么卸载了这个中断处理函数的同时会禁用该中断号。
  2. 如果指定的中断号是共享的，那么仅会删除与 dev 对应的中断处理函数。当然，如果这是删除的最后一个中断处理函数，那么也会禁用该中断号。

内核是如何实现共享的？

从上述源码可以看出，struct irq_desc *desc = irq_to_desc(irq);，首先根据中断号得到 desc 结构，然后在 __free_irq() 函数中，会根据这个中断号对应的 desc 结构所管理的 action 链表中找到与 dev_id 相对应的那个，然后删除。若是最后一个，会关掉这条中断请求线。

上述仅是对共享中断号的中断处理程序的管理，当中断触发的时候，中断号的共享，还需要硬件的支持。当中断触发，会依次调用相关中断号上的所有中断处理程序，这些中断处理程序都跟一个设备相关联，要去检查，是否是所关联设备发出的中断，如果是，则执行中断处理函数，如果不是，则直接退出。所以这里还需要硬件提供一个状态寄存器供中断处理程序读取检查，不过，大多数硬件都能提供这样的寄存器。

当一个中断处理程序在执行的时候，相应的中断号会在所有处理器上被屏蔽掉，以防止同一个中断号接收另一个新的中断。通常，这个时候其他的中断号都是打开的，其他中断能够被处理，但是当前的中断号总是被禁止。由此可以看出，同一个中断处理程序，绝对不会被同时调用以嵌套处理。

如果编写的是一个共享的中断处理程序，还应该注意：

  1. request_irq() 的参数必须设置 IRQF_SHARED 标志位。
  2. dev 参数必须唯一。
  3. 中断处理程序需要有区分时候是其所关联硬件发起中断的能力，也就是在上面提到的，应该要去读取其设备的状态寄存器，得知是否是该设备发出的中断，如果不是，需要立即退出。

以 IRQF_SHARED 标志位注册一个中断处理程序的时候，只有以下两种情况才会注册成功，一是这个中断号尚未被注册，二是之前所有注册这个中断号的所有中断处理程序都标记为了 IRQF_SHARED。

中断上下文

中断上下文与进程没有任何关系，与 current 宏也没有关系，因为没有后备进程，所以中断上下文是不能休眠的，否则的话，又如何对他重新调度呢？所以不能在中断上下文中调用会休眠的函数，这是对编写中断处理程序的限定。

中断处理程序栈的设置是一个配置选项，以前中断上下文使用的是进程的内核栈，但是 2.6 版本之后，增加了一个选项，把内核栈的大小从两页减少到了一页，然后中断栈单独占据一页。

中断处理机制的实现

  1. 设备产生中断，通过总线把电信号发送给中断控制器；

/proc/interrupts

procfs 是一个虚拟文件系统，只存在于内核内存， 一般安装于 /proc 目录。在 procfs 下读写文件都要调用内核函数，这些函数模拟从真实文件的读写。/proc/interrupts 存放的是系统中与中断相关的统计信息。

  1. 第 1 列是中断号（中断线）；
  2. 第 2 列是接收中断的统计计数器，也就是接收到的中断次数；
  3. 第 3 列是处理这个中断的中断控制器；
  4. 第 4 列是设备的 ascii 名称，也就是注册中断处理函数的第四个参数值。如果中断是共享的，那么这一列上会有以逗号+空格隔开的多个设备名字

中断控制

linux 内核提供了一套接口，用来控制机器上的中断状态。这些接口为我们提供的能力有 禁止机器上的中断系统（完全屏蔽） 和 屏蔽掉某一条中断线（屏蔽某一个中断号） 。

为什么需要这些功能呢？是为了提供同步机制。在代码里面屏蔽中断是为了防止中断处理程序抢占当前进程（不论是用户进程还是内核进程），而访问了共享资源。

上述是一个方面，另一个方面，linux 支持多处理器，linux 还提供了一套锁机制来保护共享资源。在访问共享资源之前，应该获得某种锁，使用完资源之后，解锁。这是为了保护共享资源，以免被另一个处理器同时访问。

禁止指定中断线

只针对某一条中断线上的所有设备，禁止他们传递中断信号，接口如下：

  1. disable_irq(irq) 函数需要等待 irq 指定的中断线上的所有中断处理程序退出之后才返回，返回之后该中断线被屏蔽。
  2. disable_irq_nosync(irq) 则无需等待中断处理程序的退出，直接返回，返回之后该中断线被屏蔽。
  3. synchronize_irq(irq) 等待 irq 指定的中断线上的所有中断处理程序退出之后才返回。
  4. enable_irq(irq) 应该出现在每一个 disable_irq(irq) 或者 disable_irq_nosync(irq) 之后。

有一个值得注意的点，这些函数都可以嵌套调用，并且可以在中断上下文中使用，并不会引起睡眠。如果是嵌套调用，那么要是 disable_xxx() 被调用了 n 次，那么 enable_irq() 也要调用 n 次，该中断线才是真的被激活了，否则仍然处于屏蔽状态。

上述的接口，不应该在新式的设备中使用，因为新的设备都是支持中断号共享的，如果禁止，是禁止了整个线上的所有设备。 上述的接口，在老式的设备中更可能被用到。

中断系统的状态

主要是通过一些接口来得知，自己是否处于中断上下文状态，本地中断是否被禁止。其中判断当前是否在中断上下文情况会用的比较多，因为如果要执行会引起睡眠的代码，是不能在中断上下文中执行的。

函数 说明
irqs_disabled() 如果本地中断传递被禁止，那么返回非 0 ，否则返回 0
in_interrupt() 如果内核处于任何形式的中断上下文中，返回非 0 ，要么是正在执行中断处理程序，要么是在执行中断处理程序的下半部分；否则返回 0
in_irq() 如果当前正在执行中断处理程序，返回非 0 ，否则返回 0
